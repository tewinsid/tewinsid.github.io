web编码总结

是什么

  • 字符编码：字符集中的字符编码为指定集合中某一对象（例如：比特模式、自然数序列、8位组或者电脉冲），以便文本在计算机中存储和通过通信网络的传递
  • 为什么要编码：计算机存储、传输的都是二进制流，显示需要将二进制流按照字符编码集进行转化，才能显示。

涉及知识点

常用编码

ASCII

  • ASCII字符集：主要包括控制字符（回车键、退格、换行键等）；可显示字符（英文大小写字符、阿拉伯数字和西文符号。
  • ASCII 编码：用一个字节的低 7 位表示，0~31 是控制字符如换行回车删除等；32~126 是打印字符；ASCII的最大缺点是只能解决了部份西欧语言的显示问题，但对更多其他语言依然无能为力

I字ISO-8859-X

  • ISO-8859-X 字符集：扩展的 ASCII 字符集，包括 ISO-8859-1 ~ ISO-8859-15，涵盖了大多数西欧语言字符和希腊语。
  • ISO-8859-1 编码：用 8 位表示一个字符，总共能表示 256 个字符，但还是单字节编码，不能对双字节如中日韩等进行编码。

GBXXXX

GB2312 编码

它的全称是《信息交换用汉字编码字符集 基本集》，它是双字节编码，每个汉字及符号以两个字节来表示。第一个字节称为”高位字节”（也称”区字节）”，第二个字节称为”低位字节”（也称”位字节”）。
总的编码范围是 A1-F7，其中从 A1-A9 是符号区，总共包含 682 个符号，从 B0-F7 是汉字区，它所收录的汉字已经覆盖中国大陆 99.75%的使用频率。对于人名、古汉语等方面出现的罕用字，GB2312 不能处理，这导致了后来GBK及GB 18030汉字字符集的出现。

GBK 编码

全称叫《汉字内码扩展规范》，是在 GB2312-80 标准基础上的内码扩展规范，使用了双字节编码方案，其编码范围从 8140 至 FEFE（剔除 xx7F），共 23940 个码位，共收录了 21003 个汉字，完全兼容 GB2312-80 标准，支持国际标准 ISO/IEC10646-1 和国家标准 GB13000-1 中的全部中日韩汉字，并包含了 BIG5 编码中的所有汉字。

GB18030 编码

GB18030-2005《信息技术中文编码字符集》是我国自主研制的以汉字为主并包含多种我国少数民族文字（如藏、蒙古、傣、彝、朝鲜、维吾尔文等）的超大型中文编码字符集强制性标准，其中收入汉字 70000 余个，解决了中文、日文、朝鲜语等的编码，兼容 GBK。
采用变长字节表示即单字节、双字节和四字节三种方式对字符编码。可表示27484个文字

Big5

Big5 编码： 适用于台湾、香港地区的一个繁体字编码方案。
使用了双八码存储方法，以两个字节来安放一个字，第一个字节称为”高位字节”，第二个字节称为”低位字节。

Unicode

UTF-8

采用变长字节 (1 ASCII, 2 希腊字母, 3 汉字, 4 平面符号) 表示，在网络传输中即使错了一个字节，不影响其他字节；存储效率比较高，适用于拉丁字符较多的场合以节省空间，UTF-8 没有字节顺序问题，所以 UTF-8 适合传输和通信。

UTF-16

从先前的固定宽度的 16 位编码（UCS-2）发展而来的，能够对 Unicode 的所有 1,112,064 个有效 code point 进行编码。 其编码方式是可变长度的，因为 code point 是用一个或两个 16 位代码单元编码的。在 UTF-16 文件的开头，会放置一个 U+FEFF 字符作为 Byte Order Mark（BOM）：UTF-16LE（小端序）以 FF FE 代表，UTF-16BE（大端序）以 FE FF 代表，以显示这个文字档案是以 UTF-16 编码。UTF-16 比起 UTF-8，好处在于大部分字符都以固定长度的字节 (2 字节) 储存，但 UTF-16 却无法兼容于 ASCII 编码，实际使用也比较少。

UTF-32(或 UCS-4)

对每一个 Unicode 码点使用 4 字节进行编码，其它的 Unicode 编码方式则使用不定长度编码。就空间而言，UTF-32 是非常没有效率的。尤其非基本多文种平面的字符在大部分文件中通常很罕见，以致于它们通常被认为不存在占用空间大小的讨论，使得 UTF-32 通常会是其它编码的二到四倍。虽然每一个码位使用固定长定的字节看似方便，它并不如其它 Unicode 编码使用得广泛。

涉及流程

浏览器处理

URI编码（含有非ASCII字符）

  • URI包括两部分queryString和其他，提交方式包含两种post和get
  • 浏览器根据header中的Content-Type中指定的编码形式对表单中的数据进行编码
  • URI规定：只有字母和数字[0-9a-zA-Z]、一些特殊符号”$-_.+!*’(),”[不包括双引号]、以及某些保留字，才可以不经过编码直接用于URL
Item get post
queryString 根据Content-Type编码 放到body里提交根据Content-Type编码
pathInfo 浏览器决定 浏览器决定
Content-Type设置

html中使用以下设置Content-Tyep

header编码

必然是ISO-8859-1 不能包含非ASCII字符

body编解码

body中的数据根据Content-Type中指定的字符集进行编码，如果不指定默认ISO-8859-1

规避不同浏览器问题

通过javaScript进行encodeURI() encodeURIComponent()对url中非ASCII字符进行显示编码

服务器处理

URI解码（GET含有非ASCII字符）

tomcat 在server.xml中进行配置 tomcat8之后UTF-8，之前ISO-8859-1

对应pathInfo解码
useBodyEncodingForURI 对应queryString解码

配置filte

spring有支持，实际上设置了以下两个值
request.setCharacterEncoding(encoding);
response.setCharacterEncoding(encoding);

手动设置content-Type

显示对response进行配置，与浏览器协商字符集
response.setContentType(“text/plain; charset=utf-8”)
response.setCharacterEncoding(“charset=utf-8”);

java代码

  • String类中存放的是UNICODE（UTF-16）编码
  • String s=new String(buffer,”utf-8”); 按照指定编码格式编码
  • s.getBytes(“gbk”)按照指定编码格式解码
  • OutputStreamWriter，InputStreamReader可进行编码格式的指定
  • java模拟请求中需要对uri中的中文使用URLEncoder.encode(） 进行编码

jsp相关

jsp会被处理为servlet，然后拼装html页面返回前台，jsp中通过以下标签配置content-Type，pageEncoding表示jsp文件编码

请求流程

GET
  1. URL 中 pathInfo 和 queryString 如果含有中文等非 ASCII 字符，则浏览器会对它们进行 URLEncode，编码成为 16 进制，然后从右到左，取 4 位(不足 4 位直接处理)，每 2 位做一位，前面加上%，编码成%XY 格式
  2. URL 中的 PathInfo 和 QueryString 字符串的编码和解码是由浏览器和应用服务器的配置决定，且两者编码方式不同，queryString可通过Content-Type配置
  3. 应用服务器端对 Get 请求方式解码中 pathInfo 和 queryString 的设定是不同的
  4. 比如 Tomcat 服务器一般在 server.xml 中设定的，pathInfo 部分进行解码的字符集是在 connector 的 中定义的
  5. JavaScript提供了 encodeURI()函数，它提供的是 UTF-8 的 URI 编码，java中通过 java.net.URLEncoder.encode(str,”字符集”)进行编码。
POST
  1. 浏览器根据Content-Type指定编码对表单数据进行编码

问题

web服务请求乱码

浏览器请求

  1. 配置jsp 中的page标签
  2. 前后端分离html中配置meta标签
  3. web.xml使用filter进行全局字符编码配置（spring有支持）
  4. get请求 配置tomcat以上两个配置项 server.xml
  5. 以上所有的编码格式需要统一

java代码模拟请求

  1. java使用HttpURLConnection模拟http请求
  2. http请求中url的编码通过URLEncoder.encode(zzmc, HttpTest.CHARSET)进行显示编码
  3. 不能使用String.getBytes(“gbk”)把字符串写到输出流里，一定要通过URLEncoder对非ASCII字符进行显示编码，因为URL编码结果是有%的

浏览器问题

  1. 尽量使用javascript对url进行显示编码，避免浏览器自己选择编码方式编码

硬件服务器编码问题

  1. web应用服务端与硬件服务器编码无关
