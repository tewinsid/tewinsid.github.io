java IO API 总结

是什么

在java API中，可以从其中读入一个字节序列的对象流称作输入流，而可以向其中写入一个字节序列的对象称作输出流。这些字节序列的来源和目的地可以是文件，网络连接，内存快。
java的IO操作类在包java.io下，这些类大概可以分成四组，分别是：

  1. 基于字节操作的 I/O 接口：InputStream 和 OutputStream
  2. 基于字符操作的 I/O 接口：Writer 和 Reader
  3. 基于磁盘操作的 I/O 接口：File
  4. 基于网络操作的 I/O 接口：Socket

概念

  • 流：代表任何有能力产出数据的数据源对象或者是有能力接受数据的接收端对象，流屏蔽了实际的I/O设备中处理数据的细节
  • 装饰模式:java IO api中使用装饰的方式实现对流对象的动态扩展

API

字节流

InputStream

抽象类 装饰模式中的抽象构建，一切字节输入流的父对象

ByteArrayInputStream

继承InputStream，作为装饰模式的具体构件。
适配模式：适配数组到InputStream（对象适配）

FileDescriptor

文件描述符

  1. 每个进程都有一个映射文件物理地址的表格（0标准输入，1标准输出，2标准错误，其余为进程打开过的文件地址），fileDescriptor相当于表格中的某一行数据的对应对象。
  2. 子进程继承父进程的地址表格（父进程调用fork生成子进程）
  3. andorid binder机制，a进程把文件的fileDescriptor发送给b进程，不通过集成实现。
  4. 一个打开的文件通过唯一的fileDescriptor引用，fileDescriptor是打开文件的元数据到文件物理地址的映射，在linux系统冲成为fileDescriptor，文件描述符使用一个整数表示（C语言中用int，），简写为fd
  5. 内核为每一个进程维护一个打开文件的映射文件物理地址表格，称为文件表（File table),索引是fd，数据为打开文件的信息（文件元数据，文件读取模式，指向文件inode的指针），类似与一个Map结构，子进程会默认获取父进程的File table的深拷贝，所以进程之间的file table互不影响，所以父进程子进程之间可以使用fd共享文件。
  6. fd操作实际是操作系统内核api的调用。
  7. java中的FileDescriptor类
    a. fileDescriptor提供了一个底层机器级别的数据结构，以代表打开的文件，或者socket或其他输入源
    b. 在应用中不应该创建fileDescriptor对象，fileDescriptor对象主要通过fileInputStream和FileOutputStream间接使用（适配）
    c. 在FileDescriptor中通过静态代码快和静态变量初始化标准IO
FileInputStream

继承InputStream，作为装饰器的具体构件
适配FileDescriptor到InputStream
打开文件前会进行权限检查抛出SecurityException
若fd对应文件不存在或者不能打开（目录）抛出FileNotFoundException

File

可代表文件或文件夹

FilterInputStream

继承与InputStream抽象类并内置一个InputStream域，通过调用域变量的方法实现继承来的抽象方法

BufferedInputStream

System.in 是一个BufferedInputStream
继承于FilterInputStream，是装饰模式的具体装饰类，通过内置InputStream域对象，并对域对象对应方法进行扩展实现动态扩展，支持mark

DataInputStream

继承于FilterInputStream实现了DataInput接口（），具体装饰类，内置InputStream域对象，支持以机器无关的方式从底层输入流中读取基本java类型数据。
https://docs.oracle.com/javase/7/docs/api/java/io/DataInput.html

PrintStream

System.out 和 System.err是一个PrintStream
http://www.cnblogs.com/skywang12345/p/io_16.html

字符流

（个人理解，如有问题，请校正）

  1. Reader和InputStream都是定义字符流，字节流通用接口的抽象类
  2. 定义filter主要是为了符合开闭原则，允许通过继承filter实现具体装饰类的方式动态扩展具体实现类的功能
  3. 字符流在底层实现（写入硬盘，网络）上实际还是基于字节流的，需要通过InputStreamReader，OutputSteamWriter适配对应的字节流到字符流
  4. 我理解的整个字符的api除了CharArray[Reader|Writer]、Piped[Reader|Writer]、String[Reader|Writer]等不依赖与操作系统底层支持的api，其他的接口都相当对字节流的一个装饰
  5. 所以Buffered[Reader|Writer]、InputStream[Reader|Writer]、PrintWriter不继承于Filter[Reader|Writer]，因为[Reader|Writer]本身就是一个抽象装饰类
  6. 继承于FilterReader的PushBackReader对5中隐形装饰进行二次装饰提供回退操作。
Reader

抽象类，装饰模式，实现了Readable和Closeable接口

BufferedReader

继承于Reader，为其他reader提供缓存功能
具体api详见J2SE api 文档

CharArrayReader

继承于Reader，适配char[]到Reader
具体api详见J2SE api 文档

StreamDecoder

继承于Reader，为InputStream增加编码转换功能
具体api详见J2SE api 文档

InputStreamReader

继承于Reader，适配StreamDecoder到Reader
具体api详见J2SE api 文档

Writer

与Reader类似，不详细展开
具体api详见J2SE api 文档

RandomAccessFile

  • 适用于大小已知的记录组成的文件
  • RandomAccessFile是独立的没有继承，实现了DataOutput, DataInput, Closeable接口
  • 支持在文件中前后移动
  • 本质上来说RandomAccessFile相当于把DataInputStream和DataOutputStream结合起来使用
  • 只有RandomAccessFile支持搜索方法，并且只适用于文件，
  • 在1.4以后，RandomAccessFile中的大部分功能，由nio存储映射文件所取代
  • 同时支持读写

编码问题

字节流是没有编码问题的，只有当二进制的字节流转化为字符流时需要通过特定编码方式编码、解码。

推荐使用

  • 当使用readLine()是应使用Buffered*流进行操作
  • PrintWriter可以接受Writer或者OutputStream，支持格式化输出操作
